Design Specification

EvalBench Redesign

A complete UI/UX overhaul shifting EvalBench from a cold dark dashboard to a warm, educational, and friendly evaluation workspace — one that teaches while it tests.

v1.0 Concept March 2026 Full Redesign Educational Focus

1. Design Philosophy

EvalBench has two kinds of users: practitioners who need reliable, fast evaluation tools, and learners (students, researchers) who are building intuition about what these metrics mean. The current design serves neither well — it's a cold UI shell with little educational scaffolding and a visual language that feels intimidating rather than inviting.

The redesign is built around a single principle: make evaluation feel like discovery, not busywork.

🔍

Curiosity over Complexity

Every screen should prompt "ooh, interesting!" not "what do I do here?" Use friendly copy, contextual hints, and educational callouts throughout — not just in a dedicated Learn tab.

📖

Numbers Need Context

A ROUGE-L score of 0.412 means nothing to a new user. Every metric in the interface should carry a hint of what it means — "0.412 (Excellent for 8B models)" — anchoring the number to reality.

🎓

Learn While You Work

The educational layer is not a tab you visit once. It's woven into the Eval Wizard ("Why ROUGE?"), the leaderboard (CI tooltips), and the Arena (ELO explanation). You absorb the curriculum by using the product.

☀️

Warm, Not Cold

The current blue-gray dark mode signals "hacker terminal." The redesign uses a warm off-white light mode — like a well-designed research notebook. Accessible, inviting, and credible without being sterile.

🎯

Progressive Disclosure

Start simple: the Eval Wizard presents 6 task cards. Advanced options (custom datasets, CI settings, judge configuration) are available but not visible until needed. Experts find depth; beginners find clarity.

2. Before vs. After

Here's what changes at a high level:

❌ Current Design

  • Cold dark blue-gray (#224 71% 4%)
  • Sky blue primary — feels corporate/SaaS
  • Flat nav list — no grouping or hierarchy
  • Glassmorphism cards — stylistic but low information density
  • No educational layer in the UI flow
  • Metrics shown as bare numbers with no context
  • Arena feels like a demo placeholder
  • Dashboard is a chart dump with no narrative

✅ Redesign

  • Warm off-white (#FAFAF9) — approachable, like a research notebook
  • Violet primary (#7C3AED) — academic, smart, inviting
  • Grouped nav: Evaluate / Analyze / Learn — clear mental model
  • Clean white cards with subtle shadows — information-forward
  • Inline educational tooltips, "Why this metric?" links, edu panels
  • Score badges with color grades (Excellent/Good/Fair/Poor)
  • Arena as a real battle — dramatic split, satisfying vote buttons
  • Dashboard tells a story: stat cards → leaderboard → radar → activity

Score Grading System

Every numeric score gets a color-coded grade badge so users immediately understand quality without needing domain expertise:

0.412 Excellent (≥ 0.40)
0.312 Good (0.30 – 0.39)
0.214 Fair (0.20 – 0.29)
0.091 Poor (< 0.20)

Note: Grade thresholds should be per-metric. A BLEU of 0.30 is excellent for general text but mediocre for MT. Future phase: context-aware grading per metric + task type.

Typography

Page Titles (800, 22-28px)
Section Headings (700, 16-18px)
Body text: Plus Jakarta Sans Regular, 14-14.5px, color #57534E for secondary information and descriptions across all pages.
Monospace (JetBrains Mono): All metric values, model names, latency, token counts

Key rule: All numeric scores, model identifiers, and code use JetBrains Mono. Human-readable labels, descriptions, and navigation use Plus Jakarta Sans. This dual-font system instantly communicates "this is a data value, that is a label."

5. Page-by-Page Redesign Notes

📊

Dashboard

Restructured as a narrative: stats → leaderboard → capability radar → activity. Stat cards have color-coded icons and trend arrows. The leaderboard embeds inline score bars so rankings are scannable at a glance. An educational callout at the bottom catches new users. The radar chart moves to a right-column card rather than occupying the full width — it's supporting context, not the hero.

🧪

Eval Wizard

The biggest UX improvement. Step 1 (task type) is now 6 large illustrated cards with metric chips shown inline — so users understand the implication of their choice before proceeding. An auto-configured info panel explains the chosen dataset, sample count, metrics, and estimated run time. A "Why ROUGE?" link connects directly to the Metric Guide. Steps 2-4 follow the same pattern of showing consequences before commitment.

⏳

Eval Running (Live Progress)

Replaces a blank "running…" state with a rich feedback screen: overall progress bar with shimmer animation, per-model cards with live-updating scores (prefixed with ▸ to indicate partial), and a terminal-style live log in dark-on-dark for the raw output. Users can watch scores converge in real time — which is both useful and educational.

⚔️

Arena

Redesigned as a genuine "battle" with spatial drama. The prompt gets its own violet banner at the top. The two responses are in adjacent panels separated by a minimal "VS" divider. Vote buttons are large, color-coded (blue vs. pink), and have a satisfying hover animation (lift + shadow). The hint text below each button ("More precise & structured") gives voters a nudge when unsure.

🤖

Models

Tables replaced with 3-column card grid. Each card shows a colored model icon, params/size/speed in a mini stats tray, best metric score, Arena ELO, and task tags. Un-evaluated models get a dashed border and an empty state with a CTA. The card hover lifts slightly with a violet shadow — making the interface feel alive.

📚

Metric Guide (Learn)

Rebuilt around two entry points: (1) the decision tree for users who don't know what to use, and (2) browseable metric cards grouped by task type. Each metric card shows primary vs. secondary status, a plain-English description, and when to use it. All metric cards link to the Eval Wizard pre-configured for that task — turning the guide into an on-ramp to action.

6. The Educational Layer — Design Patterns

This is EvalBench's key differentiator. The educational layer is not a separate mode or tab — it's woven into every interaction through these repeating patterns:

Pattern 1: The "?" Tooltip Badge

Every column header in the leaderboard, every metric name in a results table, gets a small circular ? badge. Hovering shows a one-sentence tooltip: "ROUGE-L: Longest Common Subsequence F1 between generated and reference text. Higher is better, max 1.0."

Implementation: Shadcn Tooltip component on a small styled span. No library needed beyond what's already installed.

Pattern 2: The Educational Callout Panel

A warm gradient panel (violet → light green) with a left border accent, used in two ways: as a welcome prompt on the dashboard ("New to LLM evaluation? Start with the Wizard…") and as a contextual explanation in the wizard ("Auto-configured for Summarization: here's what that means…").

CSS: background: linear-gradient(135deg, var(--violet-light), #F0FDF4) with a border-left: 3px solid var(--violet).

Pattern 3: Auto-Configured Feedback

In the Eval Wizard, immediately after the user selects a task type, a summary panel appears explaining: the dataset chosen, number of samples, metrics to be run, and estimated time. The user understands the full scope of what they're committing to before clicking "Continue."

This also reduces anxiety for new users — they're not clicking into the unknown.

Pattern 4: "Why this metric?" Links

Wherever a metric is mentioned in the wizard context panel, a subtle underlined link — "Why ROUGE? →" — routes to the relevant section of the Metric Guide with that metric's card highlighted. This is the core of the "learn while you work" experience.

Implementation: Simple router navigation with a query param to pre-scroll: /learn?highlight=rouge-l

Pattern 5: Score Grading + Context

Rather than showing a raw 0.412, scores display as colored badge + a context note. Phase 4 enhancement: hover a score to see a popover that says "0.412 is in the top 25% of ROUGE-L scores we've seen for 8B models on CNN/DailyMail." This turns every number into a lesson.

Requires baseline data to be stored per-dataset per-metric. Could be hardcoded initially, then computed from accumulated eval history.

7. Key Component Changes

Replacing the Current CSS Variables

The current index.css uses a cold HSL-based dark palette. The redesign replaces it with warm hex values. The variable names stay the same to minimize refactoring effort:

/* index.css — Replace existing CSS variables */ :root { /* Backgrounds (warm light mode) */ --background: #FAFAF9; /* was: 224 71% 4% (dark) */ --foreground: #1C1917; /* warm near-black */ --card: #FFFFFF; --card-foreground: #1C1917; --muted: #F5F5F4; --muted-foreground: #A8A29E; --border: #E7E5E4; /* Brand (violet replaces sky blue) */ --primary: #7C3AED; /* was: 199 89% 48% (sky blue) */ --primary-foreground: #FFFFFF; --ring: #7C3AED; /* Semantic colors */ --destructive: #DC2626; --accent: #F5F5F4; --accent-foreground: #1C1917; }

Sidebar Refactor

The sidebar changes from a glass-morphism dark component to a clean white sidebar with section labels. The active state changes from a glow effect to a solid violet-light background with violet text. Key structural additions: section label dividers, a count badge on models/history, and a "NEW" pill on the Eval Wizard link.

Score Badge Component

Add a new ScoreBadge component that accepts a value, metric type, and optionally renders the color grade:

// ScoreBadge.tsx — new component function getGrade(value: number, metric: string) { // ROUGE-L grade thresholds if (metric === 'rouge_l') { if (value >= 0.40) return 'excellent'; if (value >= 0.30) return 'good'; if (value >= 0.20) return 'fair'; return 'poor'; } // Percentage scores (accuracy, F1...) if (value >= 0.70) return 'excellent'; if (value >= 0.55) return 'good'; if (value >= 0.40) return 'fair'; return 'poor'; }

8. Implementation Roadmap

These changes are ordered from highest impact / lowest risk to most invasive. All CSS changes are in index.css or tailwind.config.ts. Component changes are additive (new components or variants) before being substitutive.

Phase A · 1–2 days CSS Only — Zero Risk

Design System Token Swap

Replace CSS variables in index.css: background to warm off-white, primary to violet, remove grid background pattern, update font weights. Update tailwind.config.ts color aliases. Result: entire app shifts to new palette without touching any component JSX.

Phase A cont. Sidebar — Low Risk

Sidebar Redesign

Add section label dividers (EVALUATE / ANALYZE / LEARN), count badges on nav items, Ollama status widget at bottom, and "NEW" pill on Eval Wizard. The active state styling changes from the current glow to solid violet-light fill. All changes in Sidebar.tsx.

Phase B · 3–5 days Component Additions — Medium Risk

Score Badges + Educational Tooltips

Add ScoreBadge component. Add MetricTooltip (the "?" badge). Replace raw score numbers in leaderboard, results table, and Run Details with ScoreBadge. Add EduCallout component (gradient panel with left border) to Dashboard and Wizard. No functional changes — purely presentational additions.

Phase B cont. Models Page — Medium Risk

Models Page: Table → Card Grid

Replace the current Models.tsx table layout with the card grid design. Cards are richer (show ELO, best score, task tags, inline stats tray) but use the same data from the existing use-models hook. Add empty state card for non-evaluated models.

Phase C · 1 week Major Page Refactors

Dashboard, Arena, and Eval Wizard Overhauls

Full structural refactor of Dashboard (stat cards + leaderboard table + right-column layout), Arena (split panel + dramatic vote buttons + prompt banner), and Eval Wizard (illustrated task cards + auto-config panel + step labels). These are the highest-impact changes and require the most QA.

9. Open Questions & Future Decisions

Dark Mode Toggle

The redesign commits to light mode as the default. Given EvalBench's developer audience, a dark mode toggle (already supported via next-themes) should be preserved. Recommend: implement a dark mode variant of the new warm palette using deep charcoal (#1C1917 inverted) rather than reverting to the old cold blue-gray.

Score Grade Thresholds

The current thresholds (Excellent ≥ 0.40 for ROUGE-L) are approximate. Final thresholds should be validated against published ROUGE baselines for CNN/DailyMail and SQuAD. Per-dataset, per-metric thresholds would be most accurate but require more data.

Mobile / Responsive

EvalBench is a local desktop tool, so mobile-first is not a priority. However, the sidebar should collapse to icons-only at viewport widths below 1100px, and cards should reflow from 3-col to 2-col to 1-col. The Arena split-panel layout needs a stacked alternative for narrow viewports.

Animation & Motion

The current app uses Framer Motion for page transitions. The redesign proposes a simpler approach: CSS animation: fadeIn for screen transitions and transition: all 0.15s for hover states. Framer Motion can be retained for the eval progress screen's shimmer and the model card's staggered entrance animation.
